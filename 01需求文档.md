# 01 需求文档 — 美食/超市商品导购助手

## 1. 文档说明

- **依据**：《记录文件.md》中的需求梳理与需求分析。  
- **范围**：本文档仅描述需求与约束，不涉及具体代码实现细节。

---

## 2. 项目概述

### 2.1 目标

开发基于 Python 的**结构化美食/超市商品导购**小程序，用于验证 **RAG（以 Chroma 为向量知识库）+ 大模型文本生成** 的完整链路可行。

### 2.2 产品形态边界

- 采用**单进程命令行（CLI）工具**即可。  
- **不提供** Web 页面、桌面 GUI 等前端形态；结果仅在**命令行输出**。

---

## 3. 功能需求

### FR-1 命令行交互与输出

| 项 | 要求 |
|----|------|
| 输出 | 导购结果在**终端（命令行）**展示。 |
| 前端 | **不需要**独立前端应用。 |

### FR-2 知识库（Chroma + RAG）

| 项 | 要求 |
|----|------|
| 向量库 | 使用 **Chroma** 作为向量存储与检索后端。 |
| RAG 流程 | 支持文档**入库（含 embedding）**；用户提问时进行 **top-k 检索**，将检索得到的片段注入 **system/user 提示词**，再交由大模型生成导购建议。 |

### FR-3 大模型接入（豆包 / 火山 Ark）

| 项 | 要求 |
|----|------|
| SDK | 使用火山引擎 **Ark** Python SDK：`volcenginesdkarkruntime`。 |
| 客户端 | 通过 `Ark(api_key=...)` 创建客户端。 |
| 鉴权 | API Key 从环境变量 **`ARK_API_KEY`** 读取。 |
| 模型 | 采用豆包模型，示例型号：**`doubao-seed-2-0-lite-260215`**（若后续变更，以实现与配置为准，但须仍为豆包/Ark 体系）。 |
| 调用方式 | 使用 **流式** 接口（`stream=True`）；对返回逐 chunk 处理，打印 **`chunk.choices[0].delta.content`**；需处理 **`choices` 为空** 及 **`delta.content` 为空** 的 chunk；流式结束后**换行**。 |

### FR-4 日志（JSON 持久化）

| 项 | 要求 |
|----|------|
| 内容 | 记录**用户输入**与**模型输出**（及实现所需的辅助信息）。 |
| 格式 | 以 **JSON** 形式写入，作为**日志**持久化保存。 |
| 建议字段 | 每次交互一条或一组 JSON 记录，至少应能支持复现与调试，例如：**时间**、**用户问题**；可选：**检索摘要**；**模型完整回复**（流式片段拼接后的结果）。 |
| 存储 | 落盘至**指定目录**或**单一日志文件**（具体路径由实现约定，须在说明中可配置或文档化）。 |

---

## 4. 业务流程（需求分析结论）

系统核心流程固定为：

1. **查库**：根据用户问题在 Chroma 中做 top-k 检索。  
2. **拼 prompt**：将检索片段注入 system/user 消息。  
3. **调 Ark**：以流式方式调用豆包模型。  
4. **终端展示**：按 chunk 输出至命令行。

---

## 5. 非功能需求与约束

- **简单可测**：优先满足验证 RAG + 大模型链路，实现保持简单。  
- **依赖可复现**：通过 `requirements.txt` 等列出 Python 依赖；通过环境变量 `ARK_API_KEY` 管理密钥。  
- **日志可追溯**：JSON 日志便于事后复现问题与对比不同版本行为。

---

## 6. 实现顺序（与《记录文件》一致）

1. 依赖与环境变量说明（含 `requirements.txt`、`ARK_API_KEY`）。  
2. Chroma 建库与示例商品/美食结构化数据入库。  
3. 检索 + Ark 调用与 CLI 主循环。  
4. JSON 日志写入。

---

## 7. 验收要点

- CLI 运行后，用户提问能在终端看到流式导购回复。  
- Chroma 中数据可被检索并参与生成。  
- 豆包通过 Ark SDK、`ARK_API_KEY` 与指定模型（或等价配置）正常流式输出。  
- 每次交互的用户输入与模型完整输出以 JSON 形式落盘可查。
